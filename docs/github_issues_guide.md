# GitHub Issue 사용 가이드

GitHub Issue를 처음 써보는 팀원을 위한 안내입니다.

---

## GitHub Issue가 무엇인가

문제, 할 일, 제안, 질문을 기록하는 공간입니다.

- 버그가 발생했을 때
- 새 기능이 필요할 때
- 문서가 부족하거나 틀렸을 때
- 나중에 해야 할 일을 기록해둘 때

---

## 이슈 만드는 순서

1. GitHub 저장소 상단 **Issues** 탭 클릭
2. 같은 내용의 이슈가 이미 있는지 검색
3. **New issue** 클릭 → 템플릿 선택 (버그 제보 / 기능 제안)
4. 제목과 본문 작성

---

## 좋은 이슈 제목

```
[버그] waypoint_nav 노드가 시뮬에서 ARM 안 됨
[버그] vtol_vision 빌드 시 cv_bridge 의존성 오류
[기능] YOLO confidence threshold 파라미터 추가
```

피하면 좋은 제목:

```
버그 있음 / 안됨 / 이거 확인 / 급함
```

---

## 버그 제보 작성 방법

- 어떤 문제가 발생했는지
- 어떻게 하면 재현되는지
- 원래 어떻게 되어야 하는지
- 관련 패키지 및 실행 환경 (시뮬/실기체)
- 터미널 로그 또는 스크린샷

---

## 이슈 ↔ PR 연결

PR 본문에 아래처럼 쓰면, PR이 병합될 때 해당 이슈가 자동으로 닫힙니다.

```text
Closes #12
Fixes #18
```

커밋 메시지에서 참조:

```text
fix: arm 재시도 로직 추가 (#12)
```

---

## 추천 작업 흐름

1. 문제 발견 / 기능 제안
2. 기존 이슈 검색
3. 새 이슈 작성
4. `develop` 기준 브랜치 생성 (예: `feature/12-yolo-threshold`)
5. 작업 후 PR 생성 → `Closes #12` 포함
6. 병합 후 이슈 자동 종료 확인
