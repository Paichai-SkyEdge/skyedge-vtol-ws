# GitHub Issue 사용 가이드

이 문서는 GitHub Issue를 처음 써보는 팀원을 위해 작성했습니다.

목표는 아래와 같습니다.

1. 언제 이슈를 만들어야 하는지 이해하기
2. 버그 / 기능 요청 / 작업 기록을 어떻게 남겨야 하는지 익히기
3. 이슈와 브랜치, 커밋, PR을 서로 연결하는 방법 익히기

## GitHub Issue가 무엇인가

GitHub Issue는 프로젝트에서 발생한 문제, 할 일, 제안, 질문을 기록하는 공간입니다.

쉽게 말하면:

- "지금 어떤 문제가 있는가"
- "무엇을 해야 하는가"
- "누가 무엇을 맡고 있는가"
- "어떤 논의가 오갔는가"

를 한곳에 모아두는 기록입니다.

## 왜 이슈를 써야 하나

이슈를 쓰면 아래 장점이 있습니다.

- 문제를 말로만 전달하지 않고 기록으로 남길 수 있음
- 나중에 왜 이런 작업을 했는지 추적할 수 있음
- 팀장이 진행 상황을 한눈에 보기 쉬움
- PR과 연결해서 어떤 문제를 해결한 코드인지 알 수 있음
- 담당자, 우선순위, 진행 상황을 정리하기 쉬움

## 언제 이슈를 만들면 좋은가

다음 상황에서는 이슈를 만드는 것을 권장합니다.

- 버그가 발생했을 때
- 새 기능이 필요할 때
- 문서가 부족하거나 틀렸을 때
- 바로 작업하지 않더라도 나중에 해야 할 일이 있을 때
- 팀 논의가 필요한 주제가 있을 때
- 테스트 중 발견한 문제를 기록해둘 때

반대로 아주 사소한 오탈자 한 줄 수정처럼 설명이 거의 필요 없는 경우는, 상황에 따라 바로 PR로 처리할 수도 있습니다.

## 이 저장소에서 사용하는 Issue 종류

현재 이 저장소에는 기본적으로 아래 템플릿이 있습니다.

- 버그 제보
- 기능 제안

즉, 보통은 아래 둘 중 하나로 시작하면 됩니다.

- 버그 제보
- 기능 제안

## 버그 제보는 언제 쓰나

아래 같은 상황에 사용합니다.

- 노드가 실행 중 죽음
- 특정 토픽이 발행되지 않음
- 시뮬레이션에서는 되는데 실기체에서는 안 됨
- Docker 빌드가 실패함
- 문서대로 했는데 실제로 동작하지 않음

예시:

- YOLO 노드 실행 시 카메라 입력이 없으면 예외 발생
- ArUco 포즈 토픽이 특정 상황에서 비어 있음
- `colcon build` 가 특정 패키지에서 실패함

## 기능 제안은 언제 쓰나

아래 같은 상황에 사용합니다.

- 새 기능이 필요함
- 기존 기능 개선이 필요함
- 새로운 모니터링 항목이 필요함
- 자동화나 CI를 추가하고 싶음
- 문서 구조를 개선하고 싶음

예시:

- Foxglove 대시보드에 배터리 경고 패널 추가
- YOLO 탐지 신뢰도 임계값 파라미터 추가
- GitHub Actions에 Docker 빌드 검증 추가

## 이슈 만드는 기본 순서

### 1. 저장소의 Issues 탭으로 이동

GitHub 저장소 상단 메뉴에서 `Issues` 를 클릭합니다.

### 2. 같은 내용의 이슈가 이미 있는지 검색

바로 새로 만들기 전에 먼저 검색해보는 것을 권장합니다.

이유:

- 같은 문제를 여러 번 등록하는 것을 줄일 수 있음
- 기존 논의를 이어갈 수 있음
- 이미 해결 중인 문제인지 확인 가능

검색 예시:

- `yolo 노드 종료`
- `aruco 포즈`
- `docker 빌드`
- `README 문서`

### 3. `New issue` 클릭

### 4. 템플릿 선택

- 버그면 `버그 제보`
- 기능 제안이면 `기능 제안`

### 5. 제목과 본문 작성

가능하면 "누가 봐도 무슨 이야기인지" 알 수 있게 씁니다.

## 좋은 이슈 제목 쓰는 방법

좋은 제목은 짧지만 구체적입니다.

좋은 예:

- `[버그] YOLO 노드가 이미지 토픽이 없을 때 종료됨`
- `[버그] px4_msgs 의존성 때문에 Docker 빌드 실패`
- `[기능] YOLO 탐지 신뢰도 임계값 파라미터 추가`
- `[기능] 초보자용 GitHub 이슈 가이드 문서 추가`

피하면 좋은 예:

- `버그 있음`
- `안됨`
- `이거 확인`
- `급함`

## 버그 제보 작성 방법

버그 이슈에는 아래 정보가 들어가면 좋습니다.

- 어떤 문제가 발생했는지
- 어떻게 하면 다시 발생하는지
- 원래는 어떻게 되어야 하는지
- 실제로는 어떻게 되었는지
- 어느 패키지와 관련 있는지
- 시뮬레이션인지 실기체인지
- 로그나 스크린샷이 있는지

현재 템플릿에도 이 항목들이 들어 있습니다.

### 버그 제보 예시

```text
제목:
[버그] YOLO 노드가 카메라 토픽이 없을 때 종료됨

버그 내용:
YOLO 노드를 실행했을 때 카메라 토픽이 연결되지 않은 경우 노드가 종료됩니다.

재현 방법:
1. 카메라 없이 yolo_detect_node 실행
2. 노드 시작 직후 로그 확인
3. 예외 발생 후 종료 확인

예상 동작:
카메라 입력이 없어도 노드가 즉시 종료되지 않고 경고 로그를 출력해야 합니다.

실제 동작:
노드가 예외와 함께 종료됩니다.

관련 패키지:
vtol_vision_yolo

환경:
- 실행 환경: 시뮬
- ROS2 버전: Humble
- Docker 사용 여부: 아니오

로그 / 스크린샷:
터미널 로그 첨부
```

## 기능 제안 작성 방법

기능 요청 이슈에는 아래 정보가 들어가면 좋습니다.

- 어떤 기능이 필요한지
- 왜 필요한지
- 어느 패키지와 관련 있는지
- 구현 아이디어가 있는지

### 기능 제안 예시

```text
제목:
[기능] 초보자용 이슈-PR 연결 가이드 문서 추가

제안 내용:
GitHub Issue를 처음 사용하는 팀원을 위해 초보자용 가이드를 추가하고 싶습니다.

필요한 이유:
현재는 문제가 생겼을 때 구두로만 전달되는 경우가 많아 기록이 남지 않습니다.

관련 패키지:
docs

구현 아이디어:
`docs/github_issues_guide.md` 문서를 추가하고 `README.md`, `CONTRIBUTING.md` 에서 링크합니다.
```

## 이슈를 잘 쓰기 위한 체크리스트

이슈를 만들기 전에 아래를 확인하면 품질이 좋아집니다.

- 제목만 봐도 대충 무슨 일인지 알 수 있는가
- 어떤 상황에서 발생하는지 적었는가
- 관련 패키지를 적었는가
- 재현 순서가 있는가
- 로그나 스크린샷을 붙일 수 있는가
- 이슈를 읽은 사람이 다음 행동을 정할 수 있는가

## 로그와 스크린샷은 어떻게 첨부하나

초보자에게 가장 도움이 되는 건 실제 증거입니다.

가능하면 아래 자료를 첨부합니다.

- 터미널 로그
- Foxglove 캡처
- 에러 메시지
- 빌드 실패 화면
- 관련 토픽 출력 결과

예시:

```bash
ros2 topic list
ros2 topic echo /vtol/aruco/pose
colcon build
```

출력 결과를 복사해서 이슈 본문에 붙이거나 스크린샷으로 첨부할 수 있습니다.

## 참고 거는 방법

GitHub에서는 다른 이슈, PR, 커밋, 파일 등을 서로 연결할 수 있습니다.

이 기능을 잘 쓰면 협업이 훨씬 쉬워집니다.

### 1. 이슈 번호로 참조하기

이슈 번호 앞에 `#` 를 붙입니다.

예시:

```text
#12
```

의미:

- 12번 이슈를 가리킴

문장 예시:

```text
이 문제는 #12 와 비슷합니다.
이번 작업은 #18 을 먼저 처리한 뒤 진행합니다.
```

### 2. PR에서 이슈 자동 닫기

PR 본문에 아래처럼 쓰면, PR이 병합될 때 해당 이슈가 자동으로 닫힙니다.

중요:

- `Closes`, `Fixes`, `Resolves` 는 GitHub가 자동으로 인식하는 예약어입니다.
- 이 부분은 자동 종료 기능 때문에 영어 그대로 사용해야 합니다.

예시:

```text
Closes #12
Fixes #18
Resolves #25
```

보통 셋 중 하나를 쓰면 됩니다.

가장 흔한 예:

```text
Closes #12
```

### 3. 커밋 메시지에서 이슈 참조하기

커밋 메시지에도 이슈 번호를 넣을 수 있습니다.

예시:

```text
fix: yolo 노드 빈 카메라 프레임 처리 추가 (#12)
docs: 초보자용 이슈 가이드 문서 추가 (#21)
```

이렇게 하면 나중에 "이 커밋이 어떤 논의와 연결되는지" 추적하기 쉬워집니다.

### 4. 파일이나 코드 위치를 언급하기

이슈 본문에 파일 경로를 적어두면 리뷰어가 이해하기 쉽습니다.

예시:

```text
관련 파일:
- src/vtol_vision_yolo/src/yolo_detect_node.py
- docs/github_issues_guide.md
```

### 5. PR 링크 붙이기

PR이 만들어지면 이슈 댓글에 아래처럼 적을 수 있습니다.

```text
해결 PR: #34
```

또는 PR 본문에:

```text
Closes #21
```

## 이슈에서 소통할 때 좋은 방식

이슈는 단순 메모장이 아니라 팀 소통 공간입니다.

그래서 아래 원칙을 권장합니다.

- 문제를 비난보다 사실 중심으로 적기
- 재현 가능한 정보 위주로 적기
- 추측과 사실을 구분해서 적기
- 진행 상황이 바뀌면 댓글로 업데이트하기
- 해결됐으면 관련 PR이나 커밋을 남기기

좋은 예:

```text
실기체 환경에서만 재현되었습니다.
현재는 시리얼 포트 초기화 시점 문제로 추정하고 있습니다.
관련 로그는 아래에 첨부합니다.
```

조금 아쉬운 예:

```text
왜 안되는지 모르겠음
아마 코드가 이상한 듯
```

## 담당자와 라벨

GitHub에서는 이슈에 담당자와 라벨을 붙일 수 있습니다.

초보 단계에서는 아래 정도만 알아도 충분합니다.

- `bug`: 버그
- `enhancement`: 기능 개선
- assignee: 누가 맡는지 표시

즉, 이슈를 보면 "무슨 종류인지"와 "누가 보고 있는지"를 알 수 있습니다.

## 추천 이슈 처리 흐름

초보자는 아래 흐름을 기본으로 사용하면 됩니다.

1. 문제 발견 또는 기능 제안
2. 기존 이슈 검색
3. 새 이슈 작성
4. 팀에서 논의
5. 작업 담당자 정리
6. `develop` 기준 작업 브랜치 생성
7. 작업 후 PR 생성
8. PR 본문에 `Closes #번호` 추가
9. 병합 후 이슈 닫힘 확인

## 이슈와 브랜치를 연결하는 이름 예시

꼭 강제는 아니지만, 브랜치 이름에 이슈 번호를 함께 넣으면 관리가 쉬워질 수 있습니다.

예시:

```text
feature/12-yolo-confidence-threshold
fix/21-aruco-pose-empty
docs/34-issue-guide
```

이 방식의 장점:

- 브랜치만 봐도 어떤 이슈 작업인지 알 수 있음
- PR과 이슈 연결이 쉬움

## PR 본문에서 자주 쓰는 문장 예시

```text
Closes #12
관련 이슈 #18
후속 작업 #21
```

의미:

- `Closes #12`: 병합되면 12번 이슈 닫기
- `관련 이슈 #18`: 관련은 있지만 자동 종료는 아님
- `후속 작업 #21`: 21번 작업의 후속 작업

## 초보자가 자주 하는 질문

### Q1. 꼭 이슈부터 만들고 작업해야 하나요?

항상 그런 것은 아닙니다.

하지만 아래 경우에는 이슈를 먼저 만드는 것이 좋습니다.

- 논의가 필요한 경우
- 작업 범위가 큰 경우
- 버그 재현 기록이 필요한 경우
- 나중에 추적이 중요할 경우

### Q2. 이슈를 만들었는데 바로 작업 못 해도 되나요?

괜찮습니다.

이슈는 "해야 할 일의 기록" 이기도 하므로, 먼저 남겨두는 것 자체가 의미 있습니다.

### Q3. 댓글만 달아도 되나요?

이미 관련 이슈가 있다면 새 이슈를 만드는 대신 기존 이슈에 댓글을 다는 편이 더 좋을 수 있습니다.

### Q4. 이슈와 PR은 뭐가 다른가요?

- 이슈: 문제 / 제안 / 할 일 기록
- PR: 실제 코드 변경을 합쳐달라는 요청

즉, 이슈는 "무엇을 해야 하는가", PR은 "어떻게 바꿨는가"에 더 가깝습니다.
